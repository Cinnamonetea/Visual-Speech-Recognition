# Анализ: MBConv vs обычный Conv2d для визуального энкодера

## 1. Контекст задачи

Модель **LipReadingTransformer** решает задачу lip reading: по видеозаписи области
губ генерирует текстовую транскрипцию. Ключевое ограничение — **15 видеоклипов**
(~960 обучающих сэмплов после скользящего окна).

Визуальный энкодер обрабатывает каждый кадр (64×96 px) покадрово, формируя
128-мерный вектор признаков, который проецируется в пространство трансформера
(d_model=64).

В ходе экспериментов мы столкнулись с двумя проблемами:
1. **Mode collapse** — декодер игнорирует визуальный энкодер и предсказывает
   самый частый токен
2. **Быстрое переобучение** — после выхода из mode collapse модель деградирует
   через 3–5 эпох

Обе проблемы связаны с тем, что обычный Conv2d backbone недостаточно эффективно
извлекает визуальные признаки при ограниченном числе параметров и данных.

---

## 2. Архитектурное сравнение

### 2.1. Обычный Conv2d (текущий backbone в model_2d.py)

```
Conv2d(3→32, k=3, s=2)  + BN + GELU    → [32, 32, 48]
Conv2d(32→64, k=3, s=2) + BN + GELU    → [64, 16, 24]
Conv2d(64→128, k=3, s=2) + BN + GELU   → [128, 8, 12]
AdaptiveAvgPool2d(1,1)                  → [128, 1, 1]
Linear(128, 64)                         → [64]
```

Каждый слой — стандартная полная свёртка (full convolution): каждый выходной канал
«смотрит» на все входные каналы одновременно.

### 2.2. MBConv backbone (Mobile Inverted Bottleneck Convolution)

```
Stem: Conv2d(3→32, k=3, s=2) + BN + SiLU         → [32, 32, 48]

MBConv(32→32,  s=1, expand=1) + SE + Skip          → [32, 32, 48]
MBConv(32→64,  s=2, expand=4) + SE + Skip(proj)    → [64, 16, 24]
MBConv(64→64,  s=1, expand=4) + SE + Skip          → [64, 16, 24]
MBConv(64→128, s=2, expand=4) + SE + Skip(proj)    → [128, 8, 12]

AdaptiveAvgPool2d(1,1)                              → [128, 1, 1]
Linear(128, 64)                                     → [64]
```

Каждый MBConv блок содержит:
1. **Expand** — pointwise Conv2d(C→C×expand_ratio), расширяет число каналов
2. **Depthwise** — Conv2d с groups=C, обрабатывает каждый канал отдельно
3. **Squeeze-Excitation** — channel attention, перевзвешивает каналы
4. **Project** — pointwise Conv2d(C×expand_ratio→out_channels), сжимает обратно
5. **Skip connection** — если in_channels == out_channels и stride == 1
6. **DropPath** — стохастическое отключение блока при обучении (регуляризация)

## 3. Почему MBConv предпочтителен для lip reading на малом датасете

### 3.1. Squeeze-Excitation: адаптивное внимание к каналам

**Проблема с Conv2d:** все каналы обрабатываются одинаково. При 128 выходных
каналах некоторые содержат шум (фон, освещение), другие — полезные признаки
(форма губ, открытие рта). Обычная свёртка не различает их.

**SE-блок решает это:**
```
x_se = GlobalAvgPool(x)              # [C, 1, 1] — «что содержит каждый канал»
x_se = Conv1x1(C → C/4) → SiLU      # сжатие
x_se = Conv1x1(C/4 → C) → Sigmoid   # расширение → веса [0, 1] для каждого канала
x = x * x_se                         # перевзвешивание
```

Результат: модель **учится автоматически подавлять шумовые каналы** и усиливать
каналы с информацией о движении губ. Это особенно ценно при малом датасете,
где модель должна максимально эффективно использовать каждый признак.

### 3.2. Skip connections: улучшение градиентного потока

**Проблема с Conv2d:** в наших экспериментах мы наблюдали, что градиент визуального
энкодера (`enc_cnn`) падает в 5–8× за первые 10 эпох. Декодер находит тривиальное
решение (mode collapse) и перестаёт «запрашивать» энкодер через cross-attention.

```
Эпоха  1: enc_cnn = 0.43    ← начальный градиент
Эпоха 11: enc_cnn = 0.088   ← упал в 5×, энкодер «отключается»
```

**Skip connections решают это:** даже если верхние слои получают слабый градиент
от cross-attention, skip connection обеспечивает **прямой путь** для градиента
к ранним слоям, минуя промежуточные. Формально:

```
# Conv2d: градиент проходит через все слои последовательно
grad = ∂L/∂x₃ · ∂x₃/∂x₂ · ∂x₂/∂x₁     ← затухает мультипликативно

# MBConv с skip: градиент имеет прямой путь
grad = ∂L/∂(x + F(x)) = ∂L/∂x + ∂L/∂F(x)  ← минимум = ∂L/∂x (не затухает)
```

### 3.3. DropPath (Stochastic Depth): встроенная регуляризация

**Проблема:** модель переобучается через 24 эпохи (val_acc: 0.139 → 0.118
за 4 эпохи).

**DropPath** случайно отключает целые MBConv блоки при обучении:
```
if random() < drop_path_rate:
    output = identity           # блок пропущен, skip connection работает
else:
    output = block(x) + identity  # нормальный forward pass
```

Это создаёт эффект **ансамбля подсетей** — при каждом forward pass активна
случайная комбинация блоков. В отличие от обычного Dropout (который отключает
отдельные нейроны), DropPath отключает целые вычислительные блоки, что является
более сильной формой регуляризации для свёрточных сетей.

В нашем конфиге: `drop_path_rate` нарастает линейно от 0 (первый блок) до 0.1
(последний блок) — ранние слои всегда активны, поздние иногда отключаются.

### 3.4. Depthwise separable convolution: больше capacity за те же параметры

**Обычная Conv2d(32→64, k=3):**
- Каждый из 64 выходных фильтров «смотрит» на все 32 входных канала
- Параметры: 32 × 64 × 3 × 3 = **18 432**
- Один тип операции: пространственная + межканальная фильтрация

**Depthwise separable (expand + depthwise + project):**
- Expand: 32→128 (pointwise, межканальное смешивание)
- Depthwise: 128 каналов × k=3×3 (пространственная фильтрация каждого канала)
- Project: 128→64 (pointwise, сжатие)

```
Параметры: 32×128 + 128×9 + 128×64 = 4096 + 1152 + 8192 = 13 440
```

При этом промежуточное представление имеет 128 каналов (а не 64), что даёт
**в 2× больше «рабочего пространства»** для извлечения признаков.

### 3.5. SiLU/Swish активация

MBConv использует SiLU (Swish) вместо GELU/ReLU:

```
SiLU(x) = x · σ(x)     где σ — сигмоида
```

SiLU является стандартом для EfficientNet-семейства моделей. Ключевое
отличие от ReLU: SiLU не обнуляет отрицательные значения полностью,
а плавно подавляет их. При малом числе каналов (32, 64) это критично —
ReLU может «убить» до 50% активаций, SiLU сохраняет информацию.

## 4. Сравнение параметров и вычислительной стоимости

### 4.1. Параметры CNN backbone

| Компонент | Conv2d | MBConv |
|---|---:|---:|
| Stem / Layer 1 | 928 | 928 |
| Layer 2 / Block 1 | 18 560 | ~2 000 |
| Layer 3 / Block 2 | 73 984 | ~18 400 |
| — / Block 3 | — | ~44 700 |
| — / Block 4 | — | ~69 600 |
| Linear(128→64) | 8 256 | 8 256 |
| **Итого CNN** | **~102 000** | **~144 000** |

MBConv backbone содержит на **~42%** больше параметров, но эти параметры
распределены по SE-блокам, skip-проекциям и expanded representations,
а не сосредоточены в нескольких больших свёрточных ядрах.

### 4.2. Полная модель (CNN + Transformer)

| Компонент | Conv2d | MBConv |
|---|---:|---:|
| CNN backbone | ~102K | ~144K |
| Transformer Encoder (2 слоя) | ~66K | ~66K |
| Text Decoder (2 слоя) | ~210K | ~210K |
| **Итого** | **~424K** | **~466K** |

Разница в полной модели: **+10%** параметров.

### 4.3. Вычислительная стоимость (FLOPs на кадр)

| Операция | Conv2d | MBConv |
|---|---:|---:|
| Layer 1 / Stem | ~2.6M | ~2.6M |
| Layer 2 / Blocks 1-2 | ~14.2M | ~22M |
| Layer 3 / Blocks 3-4 | ~14.2M | ~50M |
| **Итого на кадр** | **~31M** | **~76M** |
| **На батч (40 кадров × 12)** | **~14.9G** | **~36.5G** |

MBConv требует **~2.4× больше FLOPs** на кадр. Это связано с expand-фазой
(расширение каналов в 4×) и SE-блоком. Однако:

- Depthwise свёртки эффективно используют GPU-параллелизм
- SE-блок работает с [C, 1, 1] тензорами (минимальные вычисления)
- На практике замедление **~1.5–2×**, а не 2.4×

### 4.4. Итоговое сравнение

| Метрика | Conv2d | MBConv | Разница |
|---|---|---|---|
| Параметры CNN | 102K | 144K | +42% |
| Параметры всего | 424K | 466K | **+10%** |
| FLOPs/кадр | 31M | 76M | +145% |
| Channel attention | нет | **SE** | качественное улучшение |
| Skip connections | нет | **2 блока** | качественное улучшение |
| DropPath | нет | **0→0.1** | качественное улучшение |
| Ожидаемое замедление | — | ~1.5–2× | приемлемо |

## 5. Почему для нас это выгодный обмен

При обычном Conv2d мы получили **val_acc = 0.139** с переобучением через 24 эпохи.
Основные ограничения:

1. Градиент энкодера затухает (enc_cnn: 0.43 → 0.09 за 10 эпох)
   → **skip connections помогут**

2. Переобучение через 24 эпохи на 960 сэмплах
   → **DropPath даст дополнительную регуляризацию**

3. Энкодер не различает информативные и шумовые каналы
   → **SE attention научится перевзвешивать каналы**

4. Все эти улучшения стоят лишь **+10%** к общему числу параметров модели
   и **~1.5× замедление** обучения, что приемлемо для ~960 сэмплов.

Увеличение числа FLOPs — это не «дороже», а «богаче»: каждый кадр проходит
через более выразительную обработку (SE attention, expanded representations),
что даёт энкодеру больше шансов извлечь визуально значимые признаки из
маленьких кадров губ 64×96.
