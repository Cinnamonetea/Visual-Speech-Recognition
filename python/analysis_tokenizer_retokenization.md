# Анализ: пересборка датасета с уменьшенным словарём

## Контекст

Модель `LipReadingTransformer` обучается на 15 видеоклипах (pt1–pt15), предобработанных в `.pkl`-файлы.
Текущий токенизатор — **GPT-2**, словарь **50 259 токенов**.
Датасет маленький, и большинство токенов GPT-2 в нём никогда не встречается.

---

## Диагностика: почему обучение стоит на месте

| Метрика | Значение |
|---|---|
| val_loss (лучший) | ~7.3 |
| Test token accuracy | ~14–15% |
| Random loss для 50 259 токенов | `log(50259) ≈ 10.83` |
| Random loss для 1 200 токенов | `log(1200)  ≈  7.09` |

val_loss = 7.3 **почти равен случайному угаданию** для словаря из 1 200 токенов.
Это означает: модель не учит содержательный маппинг видео → текст, а лишь
немного превосходит случайный угадыватель для большого словаря.

### Конкретные причины

| Проблема | Объяснение |
|---|---|
| Огромная эмбеддинг-таблица | `50 259 × 64 ≈ 3.2 M параметров` — большинство строк никогда не получают градиента |
| Огромный выходной слой | `fc_out: 64 → 50 259` — модель выбирает из 50 K+ классов, видя большинство ~0 раз |
| Зависание на частых токенах | Инференс выдаёт `"the,"` вместо `"London. On tonight's program..."` |

---

## Возможность пересборки без исходных видео

**Вывод: да, возможна.** В `.pkl`-файлах уже хранятся все необходимые данные:

| Поле | Содержимое | Нужно менять? |
|---|---|---|
| `frames` | numpy-кадры губ | **Нет** |
| `num_frames` | количество кадров | **Нет** |
| `input_ids` | токены GPT-2 | **Да** → декодировать → ретокенизировать |

Алгоритм пересборки:
1. Загрузить `input_ids` из всех `.pkl` → декодировать обратно в текст через GPT-2
2. Собрать обучающий корпус из этих текстов
3. Обучить новый BPE-токенизатор (`tokenizers`, HuggingFace) на корпусе с `vocab_size ≈ 512–1 200`
4. Пересохранить `.pkl`, заменив только `input_ids` (кадры не трогаем)
5. Обновить конфиг модели: `vocab_size = new_size`

---

## Выбор размера словаря

15 клипов (похоже на новостные сюжеты типа LRS/BBC):
- Уникальных слов в корпусе: **300–600** (оценка)
- BPE с `vocab_size = 512–1 024`: покрывает 100% датасета
- `vocab_size = 1 200` — хороший выбор **с запасом**; можно начать с 512

### Почему не символьный уровень?

При character-level токенизации слово `"London"` = 6 токенов.
Текущий `max_tokens = 12` покрывал бы лишь ~2 слова.
Пришлось бы увеличивать `max_tokens` до ~60 — это ломает всю логику скользящего окна и увеличивает сложность декодера.

**Рекомендация:** subword BPE, `vocab_size ≤ 1 200`, `max_tokens = 12` оставить без изменений.

## Ожидаемый эффект

| Аспект | До | После |
|---|---|---|
| Параметры эмбеддинга | ~3.2 M | ~77 K (vocab=1200) |
| Параметры `fc_out` | ~3.2 M | ~77 K |
| Random loss | 10.83 | 7.09 |
| Покрытие токенов обучением | Частичное | 100% |
| Инференс | `"the,"` | осмысленные слова/подслова |

Уменьшение словаря **напрямую даёт модели возможность учиться**: каждый токен
встречается многократно, градиенты распределяются по реально используемым строкам
эмбеддинга, а выходной слой имеет достижимое число классов.
