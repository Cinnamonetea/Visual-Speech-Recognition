# Анализ неудачного обучения (BPE, v5)

## Результаты эксперимента

| Метрика | Значение |
|---|---|
| Лучший val_loss | ~6.00 (эпоха 79) |
| Test loss | 6.028 |
| Test token accuracy | **11.8%** |
| Предсказание на всех тестовых примерах | `"."` |

Для сравнения: случайный угадыватель для vocab=1200 даёт `log(1200) ≈ 7.09`.
Модель слегка лучше случайного по loss, но **полностью провалила инференс**.

---

## Диагностика: что пошло не так

### 1. Mode collapse — модель всегда предсказывает один токен

```
[0] Predicted:    .
[1] Predicted:    .
[2] Predicted:    .
```

Это не переобучение — это **вырожденное решение**: модель нашла минимум loss,
всегда предсказывая самый частый токен (`"."`) и игнорируя видео-вход полностью.

`val_acc = 0.115` зафиксирована с эпохи 4 до ~40 — это ровно частота токена `"."`
в валидационном наборе. Декодер научился угадывать унимодальное распределение,
не используя cross-attention к энкодеру.

### 2. Визуальный энкодер не учится

```
train_loss: 7.27 → 5.72   (падение на 1.55)
val_loss:   7.21 → 6.00   (падение на 1.21)
```

Снижение loss идёт одинаково на train и val — это признак того, что модель учится
**не на визуальных признаках**, а на статистике токенов (language prior).
Если бы энкодер реально что-то извлекал, val_loss при такой регуляризации не падал бы.

### 3. Архитектура визуального энкодера слишком слаба

Текущий pipeline извлечения признаков:

```
Conv3d(3→8, k=3³) → MaxPool3d → Conv2d(8→12, k=3, s=2) → AdaptiveAvgPool2d(1,1) → Linear(12→64)
```

- **8 каналов в 3D свёртке** — крайне мало для захвата движения губ
- **Только один Conv2d слой** с 12 каналами — backbone глубиной в 1 слой
- **AdaptiveAvgPool(1,1)** — выбрасывает всю пространственную информацию
- На выходе: `Linear(12→64)` проецирует 12-мерный вектор в пространство d_model

Итого: энкодер сворачивает кадр 64×96×3 до вектора из 12 чисел, затем
растягивает до 64. На этом информации практически нет.

---

## Предложения по улучшению

### Вариант 1: Убрать 3D свёртки, перейти на поадровый 2D CNN (рекомендуется)

**Проблема с 3D CNN:** при kernel `(3,3,3)` и только 8 каналах сеть захватывает
движение по 3 соседним кадрам, но слишком мало параметров для полезного
представления. 3D свёртки оправданы при глубоких backbone (R3D, SlowFast), у нас
же они только мешают.

**Что сделать:** убрать `Conv3d` + `MaxPool3d`, заменить на более глубокий
пофреймовый 2D CNN:

```
для каждого кадра:
    Conv2d(3→32) → BN → GELU
    Conv2d(32→64) → BN → GELU → MaxPool2d
    Conv2d(64→128) → BN → GELU
    AdaptiveAvgPool2d(1,1)  →  вектор 128-мер
→ Linear(128→d_model)
→ TransformerEncoder
```

Это даст **в 10+ раз больше признаков** на кадр без кардинального роста параметров
(все веса разделяются между кадрами).

---

### Вариант 2: Активации GELU / Swish вместо ReLU

ReLU обнуляет отрицательные активации, что при малом числе каналов (8, 12) ведёт
к разреженным и информационно бедным картам признаков.

**GELU** (используется в BERT, GPT) и **SiLU/Swish** имеют плавный переход через
ноль, сохраняют больше градиентного сигнала:

```
GELU(x) = x · Φ(x)        # Φ — CDF нормального распределения
SiLU(x) = x · σ(x)        # σ — сигмоида (Swish)
```

Изменение: в `model.py` заменить `nn.ReLU(inplace=True)` на `nn.GELU()` в CNN и
включить `activation='gelu'` в `TransformerEncoderLayer` / `TransformerDecoderLayer`.

**Ожидаемый эффект:** умеренный (+1–3% accuracy), но бесплатный с точки зрения
архитектурных изменений.

---

### Вариант 3: Взвешенная функция потерь (борьба с mode collapse)

Токен `"."` доминирует в датасете и «затягивает» модель к вырожденному решению.
Добавить веса классов в `CrossEntropyLoss`:

```python
# Подсчитать частоту токенов по train-набору
token_counts = Counter(id for _, text in file_index for id in tokenizer.encode(text).ids)
total = sum(token_counts.values())
weights = torch.ones(VOCAB_SIZE)
for token_id, cnt in token_counts.items():
    weights[token_id] = total / (VOCAB_SIZE * cnt)  # обратная частота
weights = weights.clamp(max=10.0)  # ограничить максимальный вес
criterion = nn.CrossEntropyLoss(weight=weights.to(DEVICE), ignore_index=PAD_TOKEN_ID)
```

Это штрафует модель сильнее за неверное предсказание редких токенов и не даёт
схлопнуться в `"."`.

### Вариант 4: Усиление регуляризации (дополнительно)

| Метод | Значение |
|---|---|
| Dropout в CNN | добавить `nn.Dropout2d(0.1)` после каждого Conv2d |
| DropPath (Stochastic Depth) | случайно отключать целые слои трансформера при обучении |
| Label smoothing | уже есть 0.1 — можно поднять до 0.15 |

## Рекомендуемый порядок экспериментов

| Приоритет | Изменение | Сложность | Ожидаемый эффект |
|---|---|---|---|
| 1 | Взвешенный loss (борьба с mode collapse) | низкая | устраняет `"."` из всех предсказаний |
| 2 | Заменить ReLU → GELU в CNN и трансформере | низкая | +1–3% accuracy |
| 3 | Убрать 3D CNN, сделать глубокий 2D CNN per-frame | средняя | основной прирост качества признаков |
| 4 | Увеличить каналы CNN (8→32, 12→64, добавить слой) | средняя | больше информации из кадров |

## Фундаментальное ограничение

Даже после всех улучшений архитектуры: **15 клипов — объективно мало** для
задачи lip reading. Реальный прорыв возможен только через:
- расширение датасета (хотя бы до 100+ клипов)
- предобучение визуального энкодера на публичных датасетах (LRS2, LRS3, VoxCeleb2)
- transfer learning: взять веса ResNet/EfficientNet, дообучить только трансформер
