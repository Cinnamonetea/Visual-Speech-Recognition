# Анализ: пунктуация в обучающих данных

## 1. Проблема

В обучающем датасете текстовые транскрипции содержат знаки препинания:
точки, запятые, вопросительные и восклицательные знаки, двоеточия и т.д.

Пример ground truth из текущего датасета:
```
London . O n tonight ' s program , undercover
```

BPE-токенизатор обучен на этих текстах → пунктуация попадает в словарь
и в целевые последовательности модели.

---

## 2. Почему это плохо

### 2.1. Пунктуацию невозможно предсказать по губам

Точка, запятая, вопросительный знак — это письменные соглашения, не имеющие
визуального коррелята в движении губ. Модель должна предсказывать их
«из ниоткуда», что толкает декодер полагаться на языковую статистику
вместо визуального энкодера.

### 2.2. Пунктуация забирает полезные позиции в окне токенов

При `max_tokens = 12` (11 контентных) каждый токен-знак препинания — это
потерянная позиция, которая могла бы содержать часть слова.

Пример:
```
До очистки:  London . O n tonight ' s program , undercover  (11 токенов)
После:       london on tonight's program undercover          (≤8 токенов)
```

Пунктуация забирает ~18–25% полезного окна в типичном сэмпле.

### 2.3. Пунктуация — источник mode collapse

Токен `"."` оказался самым частым в датасете (269 вхождений в train).
Модель обнаружила, что предсказание `"."` на каждой позиции минимизирует loss,
и схлопнулась в это решение. Без пунктуации в данных этой проблемы
не возникло бы.

### 2.4. Взвешенный loss не компенсирует

Взвешенный `CrossEntropyLoss` снижает вклад `"."` в градиент (weight=0.022),
но модель всё равно **обязана** предсказывать пунктуацию на правильных
позициях — это создаёт шум в обучении. Градиент от позиций с пунктуацией
не несёт визуальной информации, но обновляет веса энкодера.

### 2.5. Смешанный регистр усложняет задачу

`London` и `london` — одно слово, но разные BPE-токены. При маленьком
датасете (15 клипов) модель не может накопить достаточно примеров, чтобы
выучить оба варианта. Приведение к lowercase уменьшает словарь и
увеличивает частоту каждого токена.

## 3. Как правильно создать датасет

### 3.1. Очистка текста при декодировании из GPT-2

В `rebuild_dataset.ipynb`, при декодировании `input_ids → текст`,
добавить очистку:

```python
import re

text = gpt2_tokenizer.decode(data["input_ids"])
text = re.sub(r"[^\w\s']", '', text.lower())
corpus_texts.append(text)
```

Что делает regex `[^\w\s']`:
- Убирает: `. , ! ? ; : " ( ) — …` и любые другие знаки препинания
- Оставляет: буквы (`\w`), цифры, пробелы (`\s`), **апостроф** (`'`)
- Апостроф сохранён, чтобы не разбивать слова: `don't`, `tonight's`, `it's`
- `.lower()` приводит к единому регистру

### 3.2. Ожидаемый результат

| Метрика | До очистки | После очистки |
|---|---|---|
| Уникальных символов | ~70+ | ~30 |
| Средняя длина в токенах | ~11 | ~7–8 |
| Доля «визуально пустых» позиций | 18–25% | 0% |
| Токен `"."` в словаре | самый частый | отсутствует |
